mongodb的sharding架构搭建

sharding是一种将海量数据水平扩展的集群系统,数据分表存储在sharding的各个节点上,使用者通过简单的配置就可以很方便地构建一个分布式mongodb集群. sharding是数据库水平扩展(横向扩展或向外扩展)的解决方案,主要目的是突破单个数据库服务的I/O能力限制,最终解决数据库扩展性问题.sharding也有其弊端,比如处理事务的应用就会非常复杂.对于跨DB的事务来说,更是难保证完整性.

sharding的架构

构建一个mongodb sharding集群主要有下面3个角色:shard服务器（Shard Server），配置服务器(config Server)，路由进程（Route Process）。
Shard服务器
shard服务器即存储实际数据的分片,每个shard可以是一个mongod实例,也可以是一组mongod实例构成的Replica Sets.为了实现每个Shard内部的故障自动转换,MongoDB官方建议每个shard为一组Replica Sets.
配置服务器
Shard Key是帮助将一个collection分布在不同shard上面。Shard Key一经确认，尽量不要修改，不然容易出错。
配置服务器一般包含下面信息：

  • 每个shard节点的配置信息
  • 每个chunk的shard key范围
  • chunk在各shard的分布情况
  • 集群db与colletion的配置信息

路由进程
即mongos，可以起在客户端上，即每个端启动一个mongos，也可启动在服务器上面。它是个前端路由，客户端由此接入，首先询问配置服务器在哪个shard上操作数据，最后将数据返回客户端。客户端仅需跟路由进程（mongos）打交道，不必关心真正操作落在哪个shard上。

构架shard集群

复制集一般由一系列服务器组成，一个服务器做主库，其他服务器做从库。测试时，可以在一台服务器上面起多个实例来模拟生产环境。
数据存放路径 /data/user/mongo/data/shard/sv*
日志存放路径 /data/user/mongo/logs/shard/sv*.log

启动Shard Server

mongod --shardsvr --port 10001 --dbpath=/data/user/mongo/data/shard/sv0 --logpath=/data/user/mongo/logs/shard/sv0.log --directoryperdb
mongod --shardsvr --port 10002 --dbpath=/data/user/mongo/data/shard/sv1 --logpath=/data/user/mongo/logs/shard/sv1.log --directoryperdb

启动config Server

mongod --configsvr --port 8000 --dbpath=/var/user/mongo/data/shard/config --logpath=/data/user/mongo/logs/shard/config.log --directoryperdb

启动路由(mongos)

mongos --port 6000 --configdb localhost:8000 --logpath=/data/user/mongo/logs/shard/route.log --chunkSize 1

chunkSize用来指定chunk大小，单位MB，默认200.

gdb调试php

php进程进入死循环，php出现core文件等，是业务场景中偶尔会出现的问题。如何快速有效定位到问题所在，是我们解决问题的关键。用gdb对其进行调试，是个非常不错的手段。

调试死循环

如果php进程长期处于R状态，那么可能是进入了死循环，可通过下面示例分析：

gdb -p pid      // pid为进程id
source gdbinit  // gdbinit是php自带的一个工具
bt              // 打印堆栈信息
zbacktrace      // 代码执行过程
printzv core_globals  
printzv sapi_globals

调试core文件

线上出现core文件是一个比较严重的问题，用gdb来分析core文件是个不错的选择。调试示例如下：

gdb php -c core.222 // core.222是一个core文件
source gdbinit
bt 
p **
zbacktrace

如果手动生成core文件，可能需要设置core文件大小不受限制，linux系统下命令：ulimit -c unlimited

token设计

token作为http通信安全性方面的一个验证被广泛使用。不同的业务场景，token的设计也不同。
不过通常来说，token包含下面特性：1）必要的业务数据；2）签名串；3）自验证；4）可解析。

游戏通信的一种token设计

下面就游戏通信方面给出一个设计，直接给出php代码

/**
 * 获取通信access token
 *
 * @param string $platform 游戏平台
 * @param string $section  游戏分区
 * @param string $uid      玩家uid
 * @param int    $t        时间戳
 * @param int    $life     生命周期
 * @param string $secret   签名串需要的密钥
 * @return string
 */
function getAccessToken($platform, $section, $uid, $t, $life, $secret)
{
    $params = array(
        'platfrom' => $platform,
        'section'  => $section,
        'uid'      => $uid,
        't'        => $t,
        'life'     => $life,
    );
    $sign = getSign($params, $secret); // 该接口参照以前写的签名设计
    return sprintf("1.%s.%s.%d.%d.$d.$s", $platform, $section, $uid, $t, $life, $sign);
}

/**
 * 检测access token有效性
 * 
 * @param string $token
 * @return bool
 */
function checkAccessToken()
{
    list($type, platform, $section, $uid, $t, $life, $sign) = explode('.', $token);
    $params = array(
        'platfrom' => $platform,
        'section'  => $section,
        'uid'      => $uid,
        't'        => $t,
        'life'     => $life,
    );
    return $sign == getSign($params, $secret)
}

时效性解决

上面给出的算法，能验证token的正确性，也可以通过（$t + $life）小于当前时间戳来简单处理时效问题。但是在生命周期中，如果多点登录，任一登陆点的token都有效。
为解决这一问题，一般会采用redis或者memcache，存储每次登录的时间，验证token时，可通过$t与登录时间的比对，验证token生命周期。
上面的解决方法仍由一个缺陷，玩家单点登录，如果申请多个token，那么也会有多个有效token。我们可以在省城token时，更新缓存时间来解决问题。有人说，可以通过存储token，当重新申请token时，如果原有token有效，直接返回原token。token的自验证就是为了不存储的，所以不建议存储token。

几种加密算法

下面给出几种加密算法，有的可逆，有的不可逆，用于不同的场景。

签名

该签名算法将参数排序，加上密钥，然后计算md5值，该算法不可逆。
该算法可用于以下情况：1）网络通信，通信双方约定密钥，对于请求参数进行加密，用于验证请求正确性；2）颁发token，密钥自己保留，对方拿token请求自己的数据，用于验证权限等。
算法的php实现如下：

function getSign($params, $secret, $sign_key = 'sign')
{
    $str = '';
    ksort($params);
    foreach ($params as $k => $v) {
        if ($k != $sign_key) {
            $str .= "$k=$v";
        }
    }
    $str .= $secret;
    return md5($str);
}

id可逆加密

该算法用于id加密，算法可逆，能反解出id。该算法目的是保护真实id，将加密后id返给使用者。
该算法的php实现如下：

// 加密算法
function encodeId($id)
{
    $sid  = ($id & 0x0000ff00) << 16;
    $sid += (($id & 0xff000000) >> 8) & 0x00ff0000;
    $sid += ($id & 0x000000ff) << 8;
    $sid += ($id & 0x00ff0000) >> 16;
    $sid ^= 818329; //该值定了就不能再改了，否则就出问题了
    return $sid;
}
// 解密算法
function decodeId($sid)
{
    if (!is_numeric($sid)) {
        return false;
    }
    $sid ^= 818329; //该值定了就不能再改了，同加密算法，否则就出问题了
    $id   = ($sid & 0x00ff0000) << 8;
    $id  += ($sid & 0x000000ff) << 16;
    $id  += (($sid & 0xff000000) >> 16) & 0x0000ff00;
    $id  += ($sid & 0x0000ff00) >> 8;
    return $id;
}

对称加密

该算法可用于数据加密，是一种对称加密，能保护数据安全。
该算法的php实现如下：

// 加密
function encrypt($id, $secret)
{
    $md5_v = md5($secret);
    // Open the cipher
    $td = mcrypt_module_open('rijndael-128', '', 'cbc', '');
    // Create key and IV
    $key = substr($md5_v, 0, 16);
    $iv  = strrev(substr($md5_v, 0, 16));
    // Intialize encryption，不满16字符后面补\0
    mcrypt_generic_init($td, $key, $iv);
    // Encrypt data
    $id = mcrypt_generic($td, trim($id));
    // Terminate encryption handler and close module
    mcrypt_generic_deinit($td);
    mcrypt_module_close($td);
    return base64_encode($id);
}
// 解密
function decrypt($id, $secret)
{   
    $id = base64_decode($id);
    if (empty($id)) {
        return false;
    }
    $md5_v = md5($secret);
    // Open the cipher
    $td = mcrypt_module_open('rijndael-128', '', 'cbc', '');
    // Create key and IV，不满16字符后面补\0
    $key = substr($md5_v, 0, 16);
    $iv = strrev(substr($md5_v, 0, 16));
    // Intialize encryption
    mcrypt_generic_init($td, $key, $iv);
    // Encrypt data
    $id = mdecrypt_generic($td, $id);
    // Terminate encryption handler and close module
    mcrypt_generic_deinit($td);
    mcrypt_module_close($td);
    return trim($id);
}

redis有序集根据key排序问题

redis 有序集达不到预期排行的问题

游戏中经常用redis的有序集做排行，排行的key采用uid，排序积分用游戏数值。不同玩家，游戏数值经常相同，理想情况下，应该是早达到该数值的玩家排行在前。
由于redis有序集具有排行积分相同的，按照key进行排序的特性，我们的排序总是达不到预期的结果。
比如下面一个示例:

redis 127.0.0.1:53000> del test
(integer) 1
redis 127.0.0.1:53000> zadd test 10 1
(integer) 1
redis 127.0.0.1:53000> zadd test 10 3
(integer) 1
redis 127.0.0.1:53000> zadd test 10 2
(integer) 1
redis 127.0.0.1:53000> zrange test 0 -1 withscores
1) "1"
2) "10"
3) "2"
4) "10"
5) "3"
6) "10"

其实我们期望的排序应该是1、3、2。

排行积分中增加时间解决该问题

我们期望越早进入排行的玩家排行在前，如果用redis的有序集做排行，仅用游戏数值做排行积分无法满足我们的需求，考虑到时间因素，因而想到将时间加入的排行积分也就水到渠成。
具体的算法以下面的周排行为例：

function getRedisPoint($point)
{
    $_t    = time();
    $index = date('N', $_t);
    $begin = strtotime(date('Y-m-d', $_t));
    $min   = ($index - 1) * 1440 + ceil((SVR_NOW - $begin) / 60);
    return $point * 20000 - $min;
}

上例为计算redis周排行的一个示例，1440是一天的分钟数（60*24），一周总共10080，为计算方便采用20000，当然可以选择大于10080的任意数值。
计算真正积分的算法如下：

function getRealPoint($point)
{
    return $point > 20000 ? ceil($point / 20000) : $point;
}

其他的排行也类似，不过是选取不同的常数。

php日志监控

项目中没有对php日志进行监控，导致php fatal等处理不及时，自己写一个shell脚本，通过定时任务，对php日志进行监控，并发邮件给相应人员。

#!/bin/bash

#php日志目录
dir='php/'
log_f=$dir"php.log"
line_f=$dir"line.log"
#我们的php日志是按照天进行切割，所以需要一个记录天的文件
day_f=$dir"day.log"

today=`date +%Y%m%d`
if [ -f "$day_f" ];then
    lastday=`cat $day_f`
fi

total_l=`awk 'END{print NR}' $log_f`
if [ -f "$line_f" ];then
    last_l=`cat $line_f`
fi

if [[ "$today" -eq "$lastday" ]]; then
    read_l=$((total_l-last_l))
else    
    read_l=$total_l
fi

if [[ $read_l -gt 0 ]]; then
    echo "$total_l" >"$line_f"
    echo "$today" >"$day_f"
    content=`tail -n $read_l $log_f |grep -E "Warning|Fatal|error|PHP|#|Stack|trace|Exception|session|main|object|static|property|class|method|function|mysql|redis|mongo"`
    echo $content | mail -s 'PHP 日志监控' guanhua2011@gmail.com
fi
